クリティカルパス（Critical Path）とは、**プロジェクト全体の完了日を決定する、一連の最も時間がかかる作業経路**のこと。

この経路上の作業が1日でも遅れると、**プロジェクト全体も同じだけ遅れる**。

### 例

例えば、次のようなプロジェクトがあるとする。

| 作業     | 所要日数 | 前提条件   |
| ------ | ---: | ------ |
| A：要件定義 |   3日 | なし     |
| B：設計   |   5日 | A完了後   |
| C：機材調達 |   4日 | A完了後   |
| D：開発   |  10日 | B完了後   |
| E：テスト  |   4日 | C・D完了後 |

これを図にすると、

```
A(3日)
├─ B(5日) ─ D(10日) ─┐
└─ C(4日) ─────────┤
                    ↓
                  E(4日)
```

* A→B→D→E = **22日**
* A→C→E = **11日**

この場合、**A→B→D→E** がクリティカルパスとなる。

もしDが2日遅れれば、プロジェクト全体も2日遅れる。

一方、Cは11日で終わるため、Eが始まるまで11日間の余裕（スラック）がある。そのため、多少遅れてもプロジェクト全体には影響しない。

---

## スラック（余裕時間）

クリティカルパス上にない作業には、**スラック（Slack）**や**フロート（Float）**と呼ばれる余裕時間がある。

例

```
クリティカルパス
A → B → D → E

非クリティカル
A → C → E
```

Cには余裕があるため、数日遅れてもEの開始時刻に間に合えば問題ない。

---

## なぜ重要なのか

クリティカルパスを把握すると、

* どの作業を最優先で管理すべきか分かる
* 遅延リスクを早期に把握できる
* 人員や予算を重点的に配分できる
* 工程短縮を検討する際、どこを短縮すべきか判断できる

逆に、クリティカルパス以外の作業をいくら短縮しても、プロジェクト全体の完了日は変わらない場合が多い。

---

## 実務での活用

IT業界では、例えばシステム開発で

```
要件定義
    ↓
基本設計
    ↓
詳細設計
    ↓
プログラム開発
    ↓
結合テスト
    ↓
総合テスト
    ↓
本番リリース
```

のような流れがクリティカルパスになることが多い。

一方で、

* マニュアル作成
* 教育資料の準備
* 一部機材の手配

などは、スラックを持つことが多く、多少の遅れがあっても全体には影響しない場合がある。

---

### 要点

* **クリティカルパス**：プロジェクト全体の完了日を決める最長経路。
* **この経路上の作業は遅延が許されない**（遅れれば全体が遅れる）。
* **クリティカルパス以外の作業にはスラック（余裕時間）がある場合がある**。
* プロジェクト管理では、まずクリティカルパスを特定し、その進捗を重点的に監視することが重要。
